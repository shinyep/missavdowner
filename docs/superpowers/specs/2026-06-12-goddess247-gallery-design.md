# goddess247 图站规则设计

## 背景

当前项目的图集解析集中在 `python/image_crawler.py`。对于结构稳定、正文区域明确的站点，项目已经采用“站点识别 + 专用解析器”的方式接入，避免通用提取误抓相关推荐、侧栏缩略图或站点装饰图。

`goddess247.com` 的文章页实测特点如下：

- 图集图片直接以内嵌 `<img>` 形式出现在正文中
- 图片地址为站内 `wp-content/uploads/...` 原图地址
- 页面底部存在相关推荐缩略图，不能直接依赖全页通用 `img[src]` 抓取
- 标题可从文章 `h1` 稳定获取
- 当前样本页未发现正文图片分页，也未发现必须依赖 JavaScript 执行的图片加载

## 目标

为 `goddess247.com` 新增整站通用图集解析规则，使应用能够正确解析此类文章页的：

- 图集标题
- 正文图片列表
- 图片数量
- 预览图

并确保不会把相关推荐、站点图标、Logo、统计像素等非正文图片误加入下载列表。

## 非目标

本次不处理以下内容：

- 下载页中的网盘链接解析
- 站内搜索页、标签页、分类页解析
- 站点需要登录、验证码或额外反爬绕过的场景
- 与 `goddess247.com` 无关的通用提取器重构

## 方案选择

### 方案 A：新增站点专用解析器

做法：

- 在 `parse_gallery` 中新增 `goddess247` 站点识别
- 新增 `goddess247` 专用解析器，只针对文章正文容器抓取图片
- 标题优先取正文文章标题
- 预览图优先取正文首张图

优点：

- 能稳定避开相关推荐缩略图
- 规则风险局限在单站点，不影响现有通用站点
- 后续若该站模板轻微变化，可在单站逻辑内修补

缺点：

- 会增加少量站点专用代码

### 方案 B：增强通用图片提取器

做法：

- 修改 `_extract_image_urls` 的全局过滤逻辑
- 尝试只保留正文区域 `wp-content/uploads` 图片

优点：

- 代码改动更少

缺点：

- 容易影响其他站点
- 站点模板变化后更容易误抓
- 过滤逻辑会继续膨胀到通用路径

### 结论

采用方案 A。

## 设计

### 站点识别

新增 `goddess247` URL 识别函数，用于判断链接是否属于该站文章页。

识别条件：

- 主域名或子域名包含 `goddess247.com`

### 解析流程

`parse_gallery` 在识别到 `goddess247.com` 后，进入专用解析流程：

1. 加载页面 HTML
2. 解析文章标题
3. 定位正文主内容容器
4. 仅从正文主内容中提取图片
5. 过滤非正文图片与重复图片
6. 返回 `GalleryParseResult`

### 标题提取

优先级：

1. 文章主标题 `h1`
2. 通用标题提取兜底

标题清理沿用现有 `_clean_title`，确保与现有站点行为一致。

### 图片提取

正文图片提取规则：

- 仅在文章正文容器内查找 `img[src]`
- 仅接受站内文章图片 URL
- 保留顺序，按页面展示顺序输出
- 去重

图片过滤规则：

- 忽略 `data:` URL
- 忽略站点图标、Logo、统计像素等非图集图片
- 忽略正文外的相关推荐缩略图

### 预览图

若正文图片列表非空，则：

- `preview_base64` 保持现状，不额外生成
- 前端继续使用首张图片作为预览图来源

不额外引入 Playwright，仅沿用当前普通图片站点的 `httpx + BeautifulSoup` 路径。

## 错误处理

以下情况返回解析失败：

- 页面无法加载
- 未提取到任何正文图片

错误信息保持与现有图站一致，便于前端直接展示。

## 测试策略

本次补一个最小回归测试，覆盖：

- 能识别 `goddess247.com` URL
- 能从样本 HTML 中提取正确标题
- 只提取正文 68 张图片
- 不包含相关推荐区域的缩略图
- 第一张图片为正文首图

测试不依赖实时网络，使用固定 HTML 样本或最小化片段。

## 验收标准

当用户粘贴类似下面的文章链接时：

`https://goddess247.com/rosi%e5%86%99%e7%9c%9f-2025-10-31-no-5079-68p/`

系统应满足：

- 能成功解析标题 `ROSI写真 2025.10.31 No.5079 (68P)`
- 能提取 68 张正文图片
- 不混入相关推荐图片
- 能继续走现有下载流程

## 风险与应对

风险：

- 站点正文容器选择器未来可能变化

应对：

- 选择器采用“正文主容器优先 + 局部兜底”的方式
- 保持专用解析器边界清晰，后续修改只影响单站
